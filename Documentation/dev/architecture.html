Architecture

MQMyUI is a single MacroQuest plugin DLL. MQMyUI.cpp owns a fixed set of service singletons and a ModuleManager; each UI window is a ModuleBase subclass that reads the shared services live every frame. Multi-character sync rides a protobuf actor mailbox, settings persist to SQLite, and one global ImGui theme wraps the entire render pass.

The one big idea: modules never cache settings and never poll the game's TLOs. A single CharData::Refresh() snapshots the game each pulse, and modules pull from CharData + UiConfig every frame. This makes settings edits apply instantly and keeps per-frame game access cheap.

Module system

The module layer is intentionally thin — see Module System for the full contract and a worked "add a module" example. In brief:

  • ModuleBase (ModuleBase.h:25) — virtual hooks OnInit / OnPulse / OnRenderGUI / OnShutdown / OnZone, a bool IsEnabled, and a protected ModuleContext m_ctx. Pure-virtual GetName() returns the window/config key.
  • ModuleContext (ModuleBase.h:14) — the shared-dependency struct: Settings, Themes, Char, Chat, Actors, Icons, UI pointers, injected once per module.
  • ModuleManager (ModuleManager.h:9) — RegisterModule<T>() constructs + injects context + stores; InitAll / PulseAll / RenderAll / ZoneAll iterate enabled modules; ShutdownAll iterates all; Find (case-insensitive) and SetEnabled toggle at runtime.

Registration / render order (RegisterModules, MQMyUI.cpp:51–80)

Player → Pet → Buffs → Songs → Spells → Casting → HUD → Group → XpBars → AA → Themes → Settings → Copy → Inventory → BigBag → ITrack → Toolbar

Render order follows this list, so later modules draw on top.

Data flow

game (eqlib globals) SQLite (MQMyUI.db) | ^ v 250ms | live read each frame / write-through CharData::Refresh --> CharSnapshot --> modules <-- UiConfig (Window/Bar/Flag/Num/Str) | | v publish 1-4s render v | visibility/lock immediate; styling on Save ActorManager --> protobuf mailbox "myui" <--> peer boxes | receive v m_peers[server|char] --> Group / XPBars / Buffs / iTrack modules
  • CharData::Refresh() (CharData.cpp:81) reads pLocalPC / pLocalPlayer / zone / aggro / pAAWnd into a CharSnapshot; RefreshGroupRaid (CharData.cpp:292) scans group (≤6) and raid (≤72); RefreshBuffs / RefreshSongs walk pBuffWnd / pSongWnd slots (empty slots included for both Buffs and Songs).
  • ActorManager publishes the local snapshot as protobuf and stores received peers in m_peers keyed server|char. See Actors & Protocol.
  • Modules read live: a module never caches settings — it calls UI->Window(name), UI->Flag/Num/Str, UI->Bar(name, role) every frame, so edits apply instantly.

Theming

One global theme is applied around the whole render pass, not per-module. OnUpdateImGui calls ThemeManager::ApplyTheme(Resolved()) (which mutates ImGui::GetStyle() and returns the prior style), renders all modules, then ThemeManager::ResetTheme(saved) restores it. Resolved() returns the editor preview if the Themes editor is active, else the per-character active theme. Themes persist in the themes SQLite table. See Settings & UiConfig.
