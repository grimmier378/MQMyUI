Building

MQMyUI is built inside the MacroQuest Visual Studio solution as an external plugin. There is no separate build system, linter, or test harness — you build the DLL in VS and test it manually in-game with /plugin MQMyUI.

Toolchain

  • Visual Studio (the MacroQuest solution), C++ standard matching the MQ tree.
  • Project files: MQMyUI.vcxproj + MQMyUI.vcxproj.filters. New .cpp/.h files must be added to both.
  • Target: a plugin DLL loaded into MacroQuest at runtime.

Protobuf / abseil linking

Because MQMyUI is an external plugin, MQ2Main exports none of the protobuf runtime. The project must explicitly link all of it (x86-windows-static):

libprotobuf(d).lib
utf8_range.lib
utf8_validity.lib
absl_*.lib            (the full abseil set)
Missing or partial abseil linkage produces a wall of unresolved-external linker errors at the very end of an otherwise clean build. If a build fails only at link with absl:: symbols, the abseil lib set is incomplete — not a code problem.

The proto build step

Proto\myui.proto is compiled by MQ's ProtocolBuffer MSBuild item into Proto\myui.pb.h and Proto\myui.pb.cc, which are committed alongside the schema. Editing the .proto regenerates them on the next build. See Actors & Protocol for the message layout.

Common compile-time gotchas

These are MQ/eqlib-specific traps that show up while editing the plugin:

  • Translation units that dereference a ForeignPointer global (pLocalPC, pAAWnd, pAltAdvManager, pDBStr, pCastSpellWnd, pItemDisplayManager, pSpellDisplayManager) need #include <cassert>.
  • VePointer exposes only an explicit operator bool — test it with if/ternary/static_cast<bool>, never != nullptr.
  • The item-spell enum is ItemSpellTypes (alias eItemSpellType), not ItemSpellType.
  • mq::join does not exist in this MQ install (only split in mq/base/String.h). Prefer the MQ string helpers there over hand-rolling.
  • Headers not pulled in by <mq/Plugin.h> must be included explicitly — mq/imgui/Widgets.h, imgui/fonts/IconsFontAwesome.h, mq/base/String.h.

See Types & Constants for the full gotcha list.

Running

/plugin MQMyUI        load the plugin
/myui                 command help
/myui list            list windows + visibility
/myui <window>         toggle a window

The DB is created on first in-game pulse at {gPathConfig}/MQMyUI/MQMyUI.db. On a fresh character only the HUD and Toolbar windows appear; everything else is summoned from the Toolbar or /myui.

Live vs EMU client compatibility

The plugin compiles against both eqlib branches. The branch checked out in src/eqlib selects the client and bitness: emu-rof2 → Win32 (defines EMULATOR / IS_EMU_CLIENT=1); live → x64 (defines LIVE / IS_EMU_CLIENT=0). The vcxproj already has all four configs and imports the platform-aware Common.props, so no project edits are needed to build x64 — just build with -p:Platform=x64.

There is roughly 12 years of client divergence between RoF2 (emu) and Live. Gate divergent code with eqlib/BuildType.h macros (IS_EMU_CLIENT, HAS_*). Prefer a Get*() accessor present on both branches over an #ifdef; only guard a field/feature that exists on just one side.
