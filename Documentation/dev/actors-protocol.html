Actors & Protocol

Cross-character features (group/raid merge, XP bars, shared buffs, item tracking, remote settings, "come here"/switch) ride a single MacroQuest PostOffice mailbox named myui, carrying protobuf MyUIEnvelope messages. There are no driver/client roles — every box publishes its own state and reacts to commands addressed to it. DanNet / EQBC are not used.

box A mailbox "myui" box B CharSnapshot --publish 1-4s--> [ MyUIEnvelope ] --broadcast--> OnReceive | m_peers[server|char] OnReceive <--broadcast-- [ MyUIEnvelope ] <--publish 1-4s-- CharSnapshot | m_peers[server|char]

Receive (OnReceive, ActorManager.cpp:244)

Stores peers in m_peers[PeerKey(server,char)] (a PeerRecord), setting hasVitals/hasAA/hasBuffs and lastSeen. Self messages are ignored. When a peer is first seen (via m_seenPeers.insert(peerKey).second), a settings-request is queued via the private helper EnsurePeerSync(server, character, peerKey, peer) (ActorManager.cpp:255) regardless of which message type triggered discovery (kVitals, kAa, or kBuffs — previously the request was only sent for vitals, so a peer whose AA or buffs arrived first was silently never synced). Inventory payloads update m_itemCounts[item][char] and merge the shared tracked-item list. Command payloads dispatch to ExecuteCommand when to_char matches self. A broadcast ReloadSettings command only invokes the reload handler when this box's server|char appears in the command's arg_str (checked via ReloadTargetsSelf) — it does not reload on every broadcast.

Reload path

SetReloadHandler is wired in InitializePlugin to []{ s_settings.InvalidateCache(); s_ui.Load(); }. A setting copied/edited on another box only takes effect because InvalidateCache() clears that box's stale cache before Load() re-reads. BroadcastReload(targetList) posts one ReloadSettings command with arg_str = "srv|char;srv|char"; each receiver runs ReloadTargetsSelf and reloads only if its own name matches.

Cross-box cache pitfall: SettingsStore::SetRaw(server,char,…) only invalidates the cache when (server,char) is the current context. Writing another box's rows leaves its cache stale — which is exactly why the reload handler must call InvalidateCache() before Load().

Settings sync (SettingsSync payload)

Beyond the coarse ReloadSettings command, MQMyUI carries actual setting rows over the wire so a change made on one box can be applied to another's DB. This rides the same mailbox as a SettingsSync envelope (settings=6). Three send helpers, all on ActorManager:

HelperKindMeaning
SendSettingsPush(srv, char, rows, full)PUSH (directed)"You are the authoritative owner — apply these rows to your own DB and reply." (ActorManager.cpp:633)
BroadcastSettingsPush(rows)PUSH (undirected, to_char empty)"Mirror me" — every receiver writes the sender's rows into its local mirror copy. (ActorManager.cpp:660)
SendSettingsRequest(srv, char)REQUEST"Send me your full snapshot." Target replies with a REPLY built from GetAllSettings. (ActorManager.cpp:682)

Receive-side dispatch lives in OnReceive (ActorManager.cpp:453) and routes through two handlers wired in InitializePlugin (MQMyUI.cpp:280):

  • Apply handler (SetSettingsApplyHandler) — for a directed PUSH where we are the target: s_settings.ApplySettingRows(...) writes the rows authoritatively, then InvalidateCache() + s_ui.Load() make them take effect live; it returns the resulting current values, which are sent back as a REPLY.
  • Mirror handler (SetSettingsMirrorHandler) — for an undirected PUSH ("mirror me") or a REPLY: writes another character's authoritative values into our local mirror DB (ApplySettingRows, no UI reload). Same-store senders are skipped (the shared DB file is already current, IsSameStore).
  • A REQUEST addressed to us is answered with a full REPLY from SettingsStore::GetAllSettings.

Callers: SettingsModule sends a directed push to a chosen peer or a broadcast on demand (SettingsModule.cpp:353/369); CopyModule sends a directed push when copying a window's settings to another box (CopyModule.cpp:164). Each SettingRow is {module, name, type, value} — the same shape as a row in the settings table.

SendCommand(server, char, action, arg=0, argStr="") addresses one peer; the XP Bars allocation buttons and Group/iTrack context actions go through it.

AA's own Train/Hotkey do not use actors — they drive the local native AA window directly. Actors are only for acting on other boxes.

Protobuf schema (Proto\myui.proto)

Regenerates myui.pb.{h,cc} on build. The envelope is a oneof so a single mailbox carries every payload type.

MyUIEnvelope (oneof payload)

vitals=1 (CharVitals)  ·  aa=2 (AAInfo)     ·  buffs=3 (BuffList)
command=4 (Command)    ·  inventory=5 (InventorySnapshot)  ·  settings=6 (SettingsSync)

slot + is_empty on BuffEntry mirror empty buff slots across boxes so the group buff view lines up.

Names on the wire are always real. character carries the real name (peers use it for commands and selection keys). race=23 was added so the anonymizer is purely local: each client builds the peer's RACE_CLASS_LEVEL code from the real race/class/level on receive (PeerRecord.maskedName). Nothing masked is ever broadcast. See Types & Constants → Anonymizer.
Linking: as an external plugin, MQMyUI must explicitly link libprotobuf(d).lib + utf8_range/utf8_validity + all absl_*.lib (x86-windows-static) — MQ2Main exports none of it. See Building.
