Settings & UiConfig

All persistence lives in one SQLite DB at {gPathConfig}/MQMyUI/MQMyUI.db. SettingsStore is the low-level key/value + theme store; UiConfig is the typed per-window layer that modules read live each frame. The split matters: SettingsStore knows about rows and SQL, UiConfig knows about windows, bars, and the save/visibility policy.

Cache

A per-context m_cache keyed by module + name joined with an internal \x1F separator (implicitly the current server/character). Fetch (180) reads cache→DB; WriteRow (303) now updates the in-memory cache only after sqlite3_step() == SQLITE_DONE, so a failed write does not diverge the cache from the DB.

Cross-box pitfall: SetRaw(server,char,…) only invalidates the cache when (server,char) is the current context — so writing another box's rows leaves its cache stale. The reload handler calls InvalidateCache() before UiConfig::Load() to fix this. See Actors & Protocol.

Transactions (BeginTransaction / CommitTransaction / RollbackTransaction, lines 381–394)

Three query-wrappers for transactional batching. SaveTheme, SaveSpellSet, and SaveTrackedItems wrap their row loops in BeginTransaction()…CommitTransaction(), rolling back on a mid-loop failure (step returns non-DONE) to keep the cache and DB in sync. Previously, a failed write to row N left rows 0…N-1 committed and the cache partially updated; now the entire transaction reverts.

UiConfig (UiConfig.cpp)

Model

One WindowConfig per window holds visible, locked, textScale, iconScale plus flags<bool>, nums<float>, strs<string>, bars<BarStyle> maps (the bar map keyed by role). Modules call Window/Flag/Num/Str/Bar/Color every frame — there is no caching, so edits apply instantly. Color (line 89) and SetColor (line 90) are string-backed MQColor accessors that persist via the strs map using ColorToStr/StrToColor converters (lines 12–47).

Persistence policy

This is the core design decision of the settings layer, chosen so that dragging a slider doesn't thrash the disk:

  • Styling edits (bars, and flags/nums/strs via SetFlag/SetNum/SetStr) are in-memory until the Settings window's Save → RequestSave → Save() (a batched transaction, then Flush).
  • Visibility, lock, and active theme write through immediately (PersistVisibility / PersistField) — they're infrequent and toggled outside the Settings window.
  • Load() (404) calls SeedDefaults() then overwrites from the DB; it only reloads on character change, so zoning never wipes unsaved edits.
  • Bar fields serialize as Bar.<role>.<field> rows; colors as {r,g,b,a} strings.
Consequence: closing the Settings window without Save keeps your live changes for the session; a relog restores the last-saved styling. The Settings window's Revert button is just Load() again.

Transient visibility (IsTransientVisibility, lines 120–123)

BigBag, Inventory, iTrack, Settings, AA — PersistVisibility early-returns for these and Load() force-hides them. Rule: any window with a Toolbar toggle button belongs in this set (no persisted visibility, always starts hidden — it's summoned from the Toolbar).
